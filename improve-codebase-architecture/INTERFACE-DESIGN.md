# Interface Design（接口设计规范）
用户选定优化模块、需要调研多套备选接口方案时，启用本并行子代理工作模式。设计思路源自 Ousterhout 的「Design It Twice」理论：首个构思通常并非最优方案。

全文统一沿用 [LANGUAGE.md](LANGUAGE.md) 术语：**module**、**interface**、**seam**、**adapter**、**leverage**。

## 执行流程
### 1. 划定问题边界
拉起子代理前，先向用户说明当前待设计模块的问题范围：
- 新接口必须满足的各项约束条件
- 依赖项清单及其所属分类（参考 [DEEPENING.md](DEEPENING.md)）
- 简易示例代码片段，具象化约束规则（仅用于说明边界，不作为最终设计方案）

展示完毕后直接进入第二步，用户浏览说明的同时子代理并行执行任务。

### 2. 启动多子代理并行设计
通过 Agent 工具并行拉起 **3个及以上子代理**，每个子代理输出一套**思路完全迥异**的模块接口方案。

向每个子代理下发独立技术简报：包含文件路径、耦合细节、[DEEPENING.md](DEEPENING.md) 定义的依赖分类、Seam 内侧实现内容；简报内容和第一步面向用户的说明相互独立。
给各个代理分配差异化设计目标：
- Agent 1：精简接口，入口方法控制在1～3个，最大化单个入口的 leverage
- Agent 2：侧重拓展灵活性，兼容大量业务场景与后续扩展
- Agent 3：面向高频调用方做优化，主流使用场景做到极简调用
- Agent 4（按需启用）：基于 ports & adapters 思想设计，适配跨 Seam 依赖场景

简报内同步带入 [LANGUAGE.md](LANGUAGE.md) 架构术语与 CONTEXT.md 领域术语，保证命名和项目架构、领域定义统一。

单个子代理输出内容固定5项：
1. Interface 完整定义：类型、方法、入参 + 不变约束、调用顺序、异常分支
2. 调用示例代码，演示上层使用方式
3. Seam 背后被封装隐藏的实现内容
4. 依赖治理方案与配套 Adapter 设计（参考 [DEEPENING.md](DEEPENING.md)）
5. 方案取舍说明：标注 leverage 高收益区域与短板区域

### 3. 方案展示与横向对比
逐个展示每套接口方案便于用户理解，之后用文字做横向对比，对比维度：**depth（接口复用收益）**、**locality（变更收敛位置）**、**seam 选址**。

对比完成后给出明确选型建议：优选方案及理由；若多方案可取部分可融合，提出混合方案。结论要有明确倾向性，给用户确定性选型结论，而非罗列备选项。
