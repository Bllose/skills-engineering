# 故障排查（Diagnose）
一套针对疑难问题的标准化排查规范；若无合理理由，不得跳过任一阶段。

梳理代码时，依托项目领域术语梳理相关模块逻辑，同时查阅对应模块的ADR文档。

## 阶段1 — 搭建反馈验证链路
**本环节是整套排查的核心**，其余步骤均为落地执行。只要搭建出一套可由程序自动运行、结果明确（通过/失败）的快速校验用例，就能通过二分排查、猜想验证、埋点定位找到根因；若无校验链路，单纯阅读代码很难定位问题。

优先投入大量精力在此环节，思路大胆、灵活变通，不要轻易放弃。

### 链路搭建方案（按优先顺序尝试）
1. **失败用例**：在问题触发的代码边界编写单元/集成/e2e测试用例。
2. **Curl / HTTP脚本**：对接本地运行的开发服务发起调用。
3. **CLI调用**：传入固定测试入参，对比标准输出与基准快照。
4. **无头浏览器脚本**（Playwright / Puppeteer）：自动化操作页面，校验DOM/控制台日志/网络请求。
5. **回放抓包数据**：将真实请求、报文、事件日志落盘，在独立代码路径中复现。
6. **临时简易运行环境**：最小化拉起局部系统（单个服务+依赖Mock），单次函数调用即可触发故障。
7. **随机/模糊测试循环**：偶现异常场景，批量灌入上千组随机参数复现故障。
8. **二分排查环境**：故障出现在两个版本/数据集之间时，自动化启停对应版本，配合`git bisect run`执行二分定位。
9. **对比校验**：同一入参分别在新旧版本/两套配置执行，对比输出差异。
10. **人工介入Shell脚本**：兜底方案。必须人工点击操作时，通过`scripts/hitl-loop.template.sh`封装人工流程，实现标准化循环，采集输出用于分析。

搭建好用例链路，问题就解决九成。

### 持续优化校验链路
把校验链路当作产品迭代优化：
- 提速：缓存初始化环境、剔除无关前置逻辑、缩小测试范围。
- 精准化断言：锁定异常特征，而非仅判断“程序未崩溃”。
- 提升稳定性：固定时间、锁定RNG随机种子、隔离文件系统与网络。

单次30秒且偶发失败的链路实用性很低；2秒稳定可复现的链路是调试利器。

### 偶现随机性故障
目标不用做到100%必现，**提升复现概率**即可：循环触发上百次、多进程压测、压缩时间窗口、插入延时。复现率50%可排查，1%难以定位，持续优化直到满足调试条件。

### 确实无法搭建校验链路
明确告知用户，罗列已尝试方案，并向用户索要三类资源：(a)可复现问题的环境权限；(b)抓包HAR、日志、内核转储、带时间戳录屏等现场文件；(c)临时在线上添加埋点的授权。**没有校验链路，严禁进入猜想阶段**。

未搭建可信校验链路，不得进入第二阶段。

## 阶段2 — 稳定复现
运行校验链路，稳定触发故障。
逐项确认：
- [ ] 复现的故障和**用户描述**一致，而非临近的其他报错；问题找错，修复必然出错。
- [ ] 多次执行均可复现（随机故障需复现率达标）。
- [ ] 留存完整异常信息：报错文案、错误返回、耗时数据，用于后续验证修复效果。

未成功复现，不往下执行。

## 阶段3 — 提出故障猜想
正式验证前，整理**3~5条优先级排序的猜想**；仅靠单一猜想容易被固有思路误导。

每条猜想必须**可证伪**，附带预判结果。
> 格式：「若X是根因，则修改Y可消除故障 / 修改Z会加剧故障」

无法给出预判 = 主观猜测，舍弃或细化猜想。

**验证前把猜想清单发给用户**：用户业务信息可快速修正优先级，或告知已有被排除的猜想。用户未回复则按原有猜想继续。

## 阶段4 — 埋点观测
每条埋点只用于验证阶段3的某一条猜想，**单次只改动一个变量**。

工具选用优先级：
1. **调试器 / REPL交互式调试**（环境支持首选），一处断点优于海量日志。
2. **精准打点日志**，只在区分猜想的关键边界打印。
3. 禁止全量打日志再筛选。

**所有调试日志增加唯一标识前缀**，例：`[DEBUG-a4f2]`；收尾时通过前缀批量清理，避免调试日志残留。

**性能专项优化**：性能劣化不依赖日志，先搭建基准耗时统计（`performance.now()`、性能剖析工具、SQL执行计划），再二分定位，先量化、后优化。

## 阶段5 — 问题修复 + 编写回归用例
**修复代码前先编写回归用例**，前提是存在合理的测试切入点。

合理切入点：用例可真实还原线上触发故障的调用链路。若切入点层级过浅（单调用方用例无法复现多调用链路耦合故障），该用例无法起到防护作用。

**找不到合适切入点本身就是结论**，记录归档：代码架构导致无法编写防护用例，备注待优化。

存在合规切入点时：
1. 基于精简后的复现场景编写失败用例；
2. 运行确认用例报错；
3. 提交代码修复；
4. 再次运行用例验证通过；
5. 用原始未精简场景重新执行阶段1的校验链路。

## 阶段6 — 清理收尾 & 复盘总结
确认闭环后才算排查完成，逐项校验：
- [ ] 原始复现链路不再触发异常（重跑阶段1用例）
- [ ] 回归用例运行正常（无合适切入点则备注归档）
- [ ] 清理全部`[DEBUG-...]`调试日志（按前缀检索删除）
- [ ] 临时调试代码删除或移入专用备注目录
- [ ] PR/提交备注写明最终生效的根因猜想，方便后续维护人员参考

**收尾思考：如何从源头规避同类故障？**
若根源为架构缺陷（缺少测试切入点、调用链路混乱、隐式耦合），整理细节后调用`/improve-codebase-architecture`优化架构能力；**修复完成后再提架构优化建议**，此时掌握的信息最完整。
