# Karpathy Guidelines（卡尔帕西编码规范）
基于 Andrej Karpathy 对大模型编码弊病的总结提炼而成，用于规避LLM编码高频失误。

**取舍原则**：本规范优先稳妥而非开发速度；极简琐碎任务可灵活酌情放宽。

## 1. 动手编码前先梳理思路
**不擅自假定需求、不隐瞒疑问、主动罗列方案取舍。**

编码落地之前：
- 显性列明所有预设前提，存疑则主动问询。
- 需求存在多种解读时逐条列出，不可私自选定方案。
- 存在更简易实现方案需主动说明，不合理的需求可适度回绝。
- 内容模糊则暂停开发，点明疑点并提问确认。

## 2. 优先最简实现
**用最少代码解决问题，不提前预留未要求的扩展。**
- 不额外实现需求以外的功能。
- 仅单次使用的代码不抽取抽象封装。
- 需求未指定时，不额外预留灵活配置、可拓展设计。
- 不会出现的异常场景不编写容错逻辑。
- 若代码写了200行，但50行即可实现，需精简重写。

自我校验：资深工程师是否会判定代码过度复杂？若是则简化代码。

## 3. 精准微创改动
**只改动必要代码，仅清理自身改动带来的冗余。**

修改存量代码时：
- 不顺带优化周边无关代码、注释与代码格式。
- 不重构无故障的原有逻辑。
- 沿用项目现有编码风格，即便自身写法与之不同。
- 发现无关无效代码仅做备注，不擅自删除。

改动产生废弃冗余时：
- 仅删除因本次修改而失效的导入、变量、函数。
- 原有历史无效代码，无用户指令不得删除。

校验标准：每一行改动代码都能溯源至用户提出的需求。

## 4. 以目标驱动落地
**先定义可验证的验收标准，迭代至达标为止。**

将需求转化为可核验的落地目标：
-「补充参数校验」→「编写非法入参用例，再编码使用例通过」
-「修复BUG」→「编写可复现问题的用例，再编码修复」
-「重构X模块」→「保证重构前后全量用例均可运行」

多步骤任务先输出简要执行计划：
```
1. [步骤] → 验收：[校验项]
2. [步骤] → 验收：[校验项]
3. [步骤] → 验收：[校验项]
```

清晰的验收标准可自主迭代开发；模糊标准（如“保证能用”）需要反复确认需求。
