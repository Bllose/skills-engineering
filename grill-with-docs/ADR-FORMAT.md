# ADR 文档规范
ADR 文件统一存放于 `docs/adr/` 目录，采用顺序编号命名：`0001-slug.md`、`0002-slug.md` 以此类推。

无需提前创建目录，**懒加载生成**：首次编写 ADR 时再新建 `docs/adr/` 文件夹。

## 文档模板
```md
# {决策简短标题}

{1～3句话：说明业务背景、最终决策内容以及选型原因。}
```

规范仅此而已，一份 ADR 可以只保留一段正文。文档核心价值是**留存决策事实与决策缘由**，而非堆砌多余章节。

## 可选补充章节
仅在能产生实际价值时才添加，绝大多数 ADR 无需配置下述内容：
- **Status** 文档头配置（可选值：`proposed | accepted | deprecated | superseded by ADR-NNNN`）：适用于后续需要复审变更的决策
- **Considered Options**：仅当被否决的备选方案具备留存价值时填写
- **Consequences**：仅存在不直观的连锁负面影响、需要单独注明时填写

## 编号规则
扫描 `docs/adr/` 目录，取已有文件最大编号，在此基础上加一作为新文档编号。

## 什么时候需要编写 ADR
需同时满足以下全部3个条件：
1. **决策难以回滚**：后续推翻该方案的改造成本较高
2. **脱离上下文难以理解**：后续维护人员查看代码时，会产生「当初为什么要这么设计」的疑问
3. **经过多方权衡取舍**：存在多种可行方案，基于特定理由选定其中一种

若决策改动成本低、可以轻松回退，则无需记录；设计逻辑浅显易懂、不会产生疑问则无需记录；没有备选方案、只能采用最优解，同样无需额外存档。

### 需要落地 ADR 的场景范围
- **整体架构选型**：例如「项目采用 monorepo 架构」「写入侧采用 event-sourced，查询侧数据落地至 Postgres」
- **上下文间集成方案**：例如「Ordering 与 Billing 通过领域事件交互，不使用同步 HTTP 调用」
- **存在绑定成本的技术选型**：数据库、消息队列、鉴权服务、部署环境。不包含全部依赖库，仅记录替换周期长达一个季度的核心组件
- **领域边界与权责划分**：例如「客户数据归属 Customer 上下文，其他上下文仅通过 ID 关联引用」，明确禁止项和选用项同等重要
- **刻意背离常规方案的设计**：例如「放弃 ORM，手写原生 SQL，原因：XXX」。这类写法容易被后续开发者误优化，留存文档避免被盲目修改
- **代码无法体现的硬性约束**：例如「合规要求禁止使用 AWS」「合作方接口协议约束接口耗时必须小于200ms」
- **否决理由不直观的备选方案**：若对比过 GraphQL 最终选用 REST 且选型理由较隐晦，需要记录，避免后续人员反复提议改用 GraphQL
